Introduction:
This document provides instructions for integrating with the Access Portal system through its Application Programming Interface (API). Any third-party integrations requiring database modifications (such as adding, editing, or deleting users, tags, etc.) should exclusively carry out these operations via this designated interface. Making direct modifications to the database records will cause inconsistencies between the Access Portal web server and the SQL Server database. There are two available methods to connect to the Portal API: utilizing the lower-level TCP socket with XML or employing a .NET wrapper dll. The .NET option handles TCP socket messaging and XML to POCO conversion.
Requirements:
To utilize the API, a valid license is necessary. Three tiers of licenses are offered for selection.
Full license: 10 sessions are available.
Visitor management License: 2 sessions are available.
Time & attendance License: 1 session is available.
To manage the sessions, we have added a logout command via REST and the .Net wrapper.
The tables accessible per license are as follows. 
Full License - 10 sessions. No table limit.
Visitor Management – 2 sessions. Table access for the VisitorManagment license: ("master", "tag", "tagtype", "transack", "company", "department", "site", "profile", "masterimagetype", "masterimagelwthumb", "masterimage", "mastergroup", "masterdetailtype", "masterdetaillookup", "masterdetailgrouptype", "masterdetail", "masterasset", "accessgroup")
Time and attendance – 1 session. Table access for the T&A license ("master", "tag", "transack", "eventtype", "company", "department", "site", "area", "arealocation", "floor", "terminal", "location", "deviceimprox", "mastergroup", "masterdetailtype", "masterdetaillookup", "masterdetailgrouptype", "masterdetail")
Note: Sessions are active connected logins via the API

Connecting to API Message Server:
The code below will connect your client application to the Access Portal API Message Server.
Please be aware that the IP address must be configured to match the address of the portal application server.
Connect using XML (Java)
    try {
      // Access Portal IP address, port
      socket = new Socket("127.0.0.1", 10010);
    }
    catch (Exception e) {
      System.out.println("Invalid IP Address or Portal not running");
    }

A .NET wrapper all is available, which objectifies the API XML messages and handles the underlying TCP Socket communication to simplify the integrating to the Access Portal API.

Via RESTful Servlet
You can access the api via a RESTful servlet found at /portal/api/xml url. E.g. https://127.0.0.1/portal/api/xml.
A POST request is used for logging into Access Portal.
curl --location --request POST 'https://127.0.0.1/api/xml' \
--header 'Content-Type: application/xml' \
--header 'name: sysdba' \
--header 'pswd: bWFzdGVya2V5'
Please note that the token, which is the session required for subsequent calls, will be received upon making the REST POST call. Upon receiving a response message with headers, the token necessary for further calls will be included in these headers. It is essential to include this token in all subsequent calls.
Note: if you use 2 different URLs for example https://localhost and https://127.0.0.1 it will not work since the call used a different route.

  <apilogin login="SYSDBA" password="bWFzdGVya2V5" appname="TestApp" />

Listening for live transactions
Access Portal continually transmits live transactions gathered from the controllers.
Listen via XML (Java)
Create a thread that continually reads from a socket and waits for NULL character.
        StringBuilder sb = new StringBuilder();
        // continuously read from input stream
        while (true) {
          // if data exist in input stream, read
          if (in.available() > 0) {
            byte[] b = new byte[in.available()];
            in.read(b);
            String s = new String(b);
            while (s != null) {
              // check if null terminated
              int indexOfNull = s.indexOf("\0");
              // if not build string
              if (indexOfNull == -1) {
                sb.append(s);
                s = null;
              }
              else {
                // if null found process until null character
                sb.append(s.substring(0, indexOfNull++));
                // send received message to console
                System.out.println(sb.toString());
                sb = new StringBuilder();
                // append remaining characters for next message
                s = s.substring(indexOfNull);
              }
            }
          }
          else {
            Thread.sleep(100);
          }
        }

API XML Message
All API XML messages are wrapped within a ‘protocol’ element and are terminated by a NULL (0x00) character, which acts as a message separator. The illustration below summarises the protocol elements and its child element (commands) that can be issue to manipulate the API. It also give a brief description of their purpose. 
This protocol xsd below describes the API XML Message composition and XML parsing validation. Currently the API is lenient on the XML packet received, to assist with minimising the size and processes needed to build up a fully populated XML packet. For example, if you wish to update the Master’s Displayname, instead of building the entire validated xml with all the MasterType, MasterGroup, Site, Tags, etc., elements associated with the Master, you can just send the master ‘id’ together with the ‘displayName’ attribute and new value, to save the change.
Protocol XSD
PS. All fields and elements are case sensitive.

Add, Update or Delete Records
BaseDomain is a base object which represents a single row in any database table. All basedomains have in attribute ‘id’ which is mapped to a single composite primary key of a database table. Setting this attribute will allow the API to add, update or delete a record in thefollowing ways.
ID States
id = ‘0’ API will create a new record.
id = ‘ID’API will update the record where the ID = primary key of the record.
id = ‘-ID’API will delete the record where the ID = primary key of the record.
id = ‘null’Used to set nullable foreign keys to NULL

Best practice
Do not delete a Master record. 
Access Portal does not delete master records as they contain historical data. If you wish to remove a tagholder, set the ‘current’ attribute on the master to ‘0’. By doing this, you can set that tagholder to non-current which will remove the tagholder from the hardware and by default will not be shown in the tagholder view on Access Portal.
Keep track of replies
The protocol id is used to match the sent message with the reply. Access Portal broadcasts live transactions continually and you may have sent a message and received this broacast message first. So by checking the protocol id you will know to ignore this message.

Database Mapping table
HQL Queries
Access Portal uses Hibernate as its database mechanism, which allows us to connect to any DBMS, and as such, it has its own SQL language known as HQL. (https://docs.jboss.org/hibernate/orm/3.3/reference/en/html/queryhql.html).
Refer to the spreadsheet above to find out the database columns to hql mapping. You will be able to build up complex queries using this.
Here is an example of SQL to HQL.
Have a look at the example below and get a feel of to create HQL from SQL Query. As you can see the queries are different. The SQL statement can return any columns whereas the HQL query can on return a single domain type, so you cannot return a count in a query for example. This will probably be implemented in a future release.
EXAMPLE 1
These statements below will return the locations (doors) available in an access group with an id of 1.
SQL 
select distinct loc_name from location l join zone z on z.ZONE_ID = l.ZONE_ID join AREA_LOCATION al on al.LOCATION_ID = l.LOCATION_ID join ACCESS_GROUP ag on ag.AREA_ID = ag.ACCESS_GROUP_ID where ag.ACCESS_GROUP_ID = 1
HQL 
select distinct(l) from Location l join l.areaLocations al join al.area a join a.accessGroups ag where ag.id = 1
EXAMPLE 2
Select all available tagholder (PROFILE_ID = 16) custom fields (MASTER_DETAIL_TYPE)
SQL
select distinct MASTER_DETAIL_TYPE_ID, MDT_NAME from MASTER_DETAIL_TYPE mdt join SETTING s on 
s.SETTING_TYPE_ID = mdt.SETTING_TYPE_ID join MASTER m on m.PROFILE_ID = s.PROFILE_ID where m.PROFILE_ID = 16
HQL
select mdt from MasterDetailType mdt join mdt.settingType st join st.settings s join s.profile p where p.id=16
WARNING:
This feature is powerful but be cautious when creating queries and joining tables. Badly created joins can cause sql queries execution time to increase and impact on the performance of the Access Portal Server.
WithClause
This element was added to a few of the API XML message commands, after Access Portal v1.84 to allow reducing the amount of data being received whenever database domain object is requested. For example, you may want the tagholder information but are not interested in the images or access_groups related to that master, which normally uses a lot more bandwidth. By changing the withClause you will be able to request only what you wish to process on the client application. This element is for use with the dbsearch, luceneSearch, dbupdate, or registering as a broadcast listener which return database domain objects. The rules are as follows:
If the withClause is not set, the domain object will return all the domain’s fields as well as the foreign key domains and their fields. If a withClause element is set, it will return only the returned domains field and base on the withClause, will try do an hql lookup on those withClause values. 
Example:
If you want to return a master domain with only its tag information, add withClause of (tags), looks as follows:
<?xml version="1.0" encoding="utf-8" standalone="yes"?>
<protocol xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" id="2" version="0" xmlns="http://www.identisoft.net/protocol">
  <dbsearch id="0" tableName="Master" byId="1" recordCount="100">
    <withClause>tags</withClause>
  </dbsearch>
</protocol>

Broadcast Transaction filtering
By default, you will receive any live transactions that occur on the System as a broadcast transaction message. This causes issues when sending an API message, and a broadcast transaction was received while waiting for a response. As of Access Portal V3.04, there is an ability to switch off transaction broadcasting when connecting to the API, by setting the following properties in the CONFIGURATION table:
CONFIGURATION.CONFIG_KEY = ‘api.broadcast.transaction’ and CONFIGURATION.CONFIG_VALUE=’false’
To allow the API connection to receive these transaction, send a request live transaction when an API connection is established. To filter the received transactions so that you do not receive messages from another controller, for example, you can add a data element with key= ‘filter’ and a value of ‘controller=5’. 
Example: Only receive allowed transaction from these doors (LOCATION.LOCATION_ID=1 and LOCATION.LOCATION_ID=2)
<?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<protocol id="1" version="1.0" xsi:schemaLocation="http://www.identisoft.net/protocol protocol.xsd" xmlns="http://www.identisoft.net/protocol" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
    <command type="22">
        <data key="filter" value="eventType.etName='Allowed'"/>
        <data key="filter" value=" location.id=1,2"/>
    </command>
</protocol>
Explanation:
Hql selection
These are the table join and column matches from the transack entity, that need to be compared. The example will match transack->eventtype.etName and transack->location.id 
Match
Comma separated values to match database record against. 
The match is done by an Object.toString().equals by default, between database record and the match value.
If the value is wrapped in a single quote (‘) a case insensitive String.contains match is done between the record in the database and the match provided. 
If the value is wrapped in a two single quote (‘‘) a case sensitive String.contains match is done between the record in the database and the match provided.
Important:
An AND comparision is done between different Hql selections and an OR is done between comma separated match values.
